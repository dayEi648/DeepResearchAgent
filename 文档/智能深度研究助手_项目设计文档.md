# 智能深度研究助手（Deep Research Agent）

## 一、项目概述

**一句话描述**：用户输入一个研究主题，系统自动拆解任务 → 并行搜索收集 → 分析整合 → 生成结构化报告，全程可观测、可中断恢复。

**项目定位**：面向实习/求职的简历级项目，聚焦 LangGraph 多 Agent 协作编排、MCP 外部工具调用、RAG 知识增强与 Redis 状态持久化，完整覆盖 Agent 开发全链路。

**预期周期**：核心功能 2 周跑通，第 3 周调优 + 补文档与 Demo 视频。

---

## 二、业务主要功能

### 2.1 核心流程

1. **主题接收**：用户通过 Web 接口提交研究主题（如 "2026 年 AI Agent 开发框架对比调研"）。
2. **任务规划（Planner Agent）**：LLM 将主题拆解为若干并行的子研究任务（如 "框架生态对比"、"性能基准测试"、"社区活跃度分析"）。
3. **并行搜索（Researcher Agent × N）**：每个子任务通过 MCP 调用阿里云 IQS Search 收集资料，结果写入共享状态。
4. **分析整合（Analyzer Agent + RAG）**：
   - 先通过 RAG 检索"研究方法论知识库"，获取技术调研报告的标准结构。
   - LLM 基于搜索结果与方法论模板，生成分析结论。
5. **报告生成（Writer Agent）**：LLM 撰写 Markdown 格式的结构化研究报告。
6. **质量审核（Reviewer Agent）**：LLM 审核报告完整性、逻辑性与格式规范。
   - **通过**：流程结束，保存报告。
   - **不通过**：回退到 Writer Agent 重写（最多循环 2 次，防止死循环）。
7. **结果交付**：报告通过 Python 原生异步文件写入（aiofiles）保存到本地作为兜底，同时上传到阿里云 OSS 生成可下载外链，通过 SSE 流式推送给前端展示。

### 2.2 辅助功能

- **断点续传**：长任务执行过程中若服务重启，可从 Redis Checkpoint 恢复状态，继续执行。
- **搜索缓存**：相同查询在 1 小时内直接走 Redis 缓存，降低 API 成本与延迟。
- **并发限流**：基于 Redis Sliding Window 限制同时执行的研究任务数，防止资源耗尽。
- **进度可视化**：SSE 实时推送各 Agent 节点执行状态（Planner → Researcher → Analyzer → Writer → Reviewer）。

## 五、LangGraph 状态机详解

### 5.1 状态定义（TypedDict）

```python
class ResearchState(TypedDict):
    topic: str                # 用户输入的研究主题
    plan: List[str]           # Planner 拆解出的子任务清单
    search_results: dict      # key=子任务, value=搜索结果列表
    analysis: str             # Analyzer 的分析结论
    report: str               # Writer 生成的 Markdown 报告
    review_feedback: str      # Reviewer 的审核意见
    task_id: str              # 任务唯一标识（由 FastAPI 层传入）
    iteration_count: int      # 审核循环计数器（防死循环）
    current_node: str         # 当前执行节点名（用于 SSE 推送）
```

### 5.2 节点与职责

| 节点 | 职责 | 关键技术点 |
|------|------|-----------|
| **planner** | LLM 将 topic 拆解为 3-5 个并行的子研究问题 | Prompt Engineering、结构化输出 (JSON mode) |
| **researcher** | 对每个子问题调用 MCP Search，并行执行 | MCP Client、异步并发 (asyncio.gather) |
| **analyzer** | RAG 检索方法论 → LLM 分析整合所有搜索结果 | ChromaDB 向量检索、上下文拼接 |
| **writer** | 基于 analysis 撰写 Markdown 报告 | 长文本生成、格式控制 |
| **reviewer** | LLM 从完整性、逻辑性、格式三方面审核报告 | 自我反思 (Self-reflection) Prompt |

### 5.3 边与流转逻辑

```
START ──▶ planner ──▶ researcher ──▶ analyzer ──▶ writer ──▶ reviewer
                                                          ▲      │
                                                          │      │ 不通过且 iteration < 2
                                                          └──────┘

                                                          │      │ 通过 或 iteration >= 2
                                                          ▼      ▼
                                                        END / save_report
```

- **普通边**：planner → researcher → analyzer → writer → reviewer（顺序执行）
- **条件边（Conditional Edge）**：reviewer 之后
  - 若 feedback 包含"通过" → 流向 `save_report` → END
  - 若 feedback 包含"不通过"且 `iteration_count < 2` → 回退到 writer
  - 若 `iteration_count >= 2` → 强制 END（防止无限循环）

### 5.4 持久化配置

```python
from langgraph.checkpoint.redis import RedisSaver

# Redis 作为 Checkpoint Store
checkpointer = RedisSaver(redis_url="redis://localhost:6379/0")

# 编译图时注入持久化器
graph = builder.compile(checkpointer=checkpointer)

# 每个任务拥有独立的 thread_id（即 task_id）
config = {"configurable": {"thread_id": task_id}}
graph.ainvoke(initial_state, config=config)
```

## 六、Redis 三重角色设计

### 6.1 Checkpoint 持久化（核心亮点）

- **Key 格式**：`langgraph:checkpoint:{thread_id}`
- **作用**：LangGraph 每完成一个节点，自动将 `ResearchState` 快照写入 Redis。
- **价值**：服务重启后，通过相同 `thread_id` 调用 `graph.get_state(config)` 即可恢复，实现**断点续传**。

### 6.2 搜索结果缓存

- **Key 格式**：`search:cache:{hash(query)}`
- **TTL**：3600 秒（1 小时）
- **作用**：相同搜索词在 1 小时内直接返回缓存内容，避免重复调用 IQS Search API，降低成本与延迟。

### 6.3 并发限流

- **Key 格式**：`rate_limit:research:{window}`
- **算法**：Sliding Window Counter
- **作用**：限制单 IP / 全局同时执行的研究任务数（如最多 5 个并发），防止外部 API 配额耗尽。

## 七、RAG 知识库设计

### 7.1 设计理念

**不做通用网页爬虫，而是构建"研究方法论知识库"**，用于规范 LLM 的输出质量。

### 7.2 数据来源（手动整理 5-10 份）

- 优质技术调研报告模板（如 Google 技术文档风格）
- 竞品分析框架（如 SWOT、波特五力简化版）
- 行业白皮书目录结构示例
- 技术博客中的"如何写好技术方案"文章

### 7.3 处理流程

1. **切分**：LangChain `RecursiveCharacterTextSplitter`（chunk_size=500, overlap=50）
2. **向量化**：阿里云百炼 `text-embedding-v4`（默认 1024 维，免费额度） → 存入 ChromaDB
3. **检索**：Analyzer 节点执行 `similarity_search(query="技术调研报告结构", k=3)`
4. **增强**：将检索到的模板结构注入 Prompt，引导 Writer 按标准格式输出

**色彩哲学**：整个界面的彩色预算只花在 Apple Blue（`#0071e3`）上。深色背景用纯黑 `#000000`，浅色背景用 `#f5f5f7`（带微蓝灰的浅灰，避免纯白刺眼）。**不使用任何纹理、渐变、图案**——solid colors only，这是 Apple 的核心规则。

#### 空间与布局（Spatial Composition）

- **基础单位**：8px。
- **最大内容宽度**：980px，居中。
- **区块节奏**：纯黑 (`--bg-black`) 与浅灰 (`--bg-light`) 严格交替，形成 Apple 标志性的"电影式场景切换"。首屏纯黑 → 进度页浅灰 → 报告页白色。
- **Whitespace 哲学**：**压缩内部，扩张外部**——文字块内部紧凑（负字距、紧行高），但四周留白极其慷慨。每个主要区块接近满屏高度。不是"非对称打破网格"，而是 Apple 式的**对称居中 + 大量呼吸空间**。
- **布局结构**：对称居中为主。进度页采用单栏居中（最大 980px），而非多栏非对称。

#### 深度与质感（Depth & Texture）

- **导航玻璃条**：`backdrop-filter: saturate(180%) blur(20px)` + `background: rgba(0, 0, 0, 0.80)`，48px 高度，固定顶部。这是 Apple 最具识别度的 UI 元素。**无论下方区块是黑是白，导航条始终保持深色半透明玻璃**。
- **卡片与表面**：浅色页面上使用白色 (`#ffffff`) 卡片，圆角 8px-12px，**无边框**（Apple 极少用可见边框），**无阴影或仅使用 `--shadow-soft`**。大部分层级通过背景色对比实现（白色卡片放在 `#f5f5f7` 上）。
- **无纹理、无渐变、无图案**： backgrounds are solid colors only。不添加噪点、渐变网格、几何图案——这些会分散用户对研究内容的注意力。
- **无自定义光标**：使用系统默认光标。Apple 从不自定义光标。

#### 动效系统（Motion）

| 场景 | 动效 | 实现方式 |
|------|------|---------|
| 页面加载 | 内容 staggered fade-in + translateY(24px→0) | CSS animation，各元素延迟 100ms 递增，duration 600ms，ease-out |
| 新节点激活 | 当前步骤文字颜色从灰色平滑过渡为黑色 + 蓝色指示条从左滑入 | CSS transition 400ms |
| 新日志到达 | 列表项从透明淡入（opacity 0→1），无位移 | CSS transition 300ms |
| 按钮悬停 | 背景色微妙提亮（`#0071e3` → `#0077ed`），无发光、无阴影扩散 | CSS transition 200ms |
| 页面切换 | 旧视图淡出（opacity 1→0, 200ms），新视图从下方淡入 | Vue `<Transition>` + CSS |
| 任务完成 | 进度条平滑填满，步骤条全亮，一个 subtle 的淡入提示"报告已生成" | CSS transition sequence |

**原则**：Apple 的动效是** invisible 的**——用户感受到"流畅"，但不会被动效本身吸引注意力。所有动画 subtle、克制、服务于信息层级。

**智能深度研究 Agent 系统**

### 项目描述
- 基于 **LangGraph** 设计多 Agent 协作状态机，实现"任务规划 → 并行搜索 → 分析整合 → 报告生成 → 质量审核"的自动化深度研究流程，支持审核不通过时的循环回退机制，最多重试 2 次。

- 基于 **FastAPI** 构建异步服务，支持 SSE 流式推送 Agent 各节点实时执行进度，具备高并发任务提交与状态查询能力。

## 十三、已知风险与建议

> 以下内容来自代码审查，开发前请阅读，避免踩坑。

### 风险清单

| # | 风险点 | 说明与应对 |
|---|--------|-----------|
| 1 | **RedisSaver 非官方内置** | `langgraph.checkpoint.redis` 不是 LangGraph 官方自带实现，需安装第三方包（如 `langgraph-checkpoint-redis`）或自行继承 `BaseCheckpointSaver` 实现。不影响功能，提前装好包即可。 |
| 2 | **Researcher 并行写 State** | 3 个 Researcher 并行执行时同时往 `search_results` 写入，可能出现竞态。实现时需使用 LangGraph 的 `Send` 或 map-reduce 机制合并结果。 |
| 3 | **MCP Server 连接** | IQS Search MCP Server 是远程 SSE 服务，无需本地下载安装。项目通过 `mcp_settings.json` 配置连接参数，Python 侧管理 SSE 长连接生命周期。 |
| 4 | **错误处理缺失** | 设计文档未覆盖搜索超时、LLM API 限流、RAG 检索为空等异常。实现时各节点需补充 `try/except` 与降级策略，防止单点故障拖垮整个流程。 |
| 5 | **前端状态管理未指定** | 三视图 SPA 涉及 `task_id` 跨页面传递、SSE 断线重连、日志列表维护，建议引入 **Pinia** 做全局状态管理。 |

### 实现建议

1. **节点超时**：每个 Agent 节点设置 LLM 调用超时（如 60 秒），防止外部 API 阻塞导致任务挂起。
2. **健康检查**：FastAPI 增加 `GET /health` 接口，方便快速验证服务与依赖（Redis、ChromaDB）是否就绪。
3. **任务队列**：用 Redis List 实现简单后台队列，避免长时运行的 LangGraph 任务直接占用 FastAPI 请求连接。
4. **SSE 重连**：前端 `EventSource` 在 `onerror` 事件中做延迟重连（如 3 秒后重新连接），提升弱网环境下的稳定性。
